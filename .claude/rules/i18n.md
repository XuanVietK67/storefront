# i18n Rules

Rules for multi-language support. Apply automatically on every feature and bug fix — do not wait to be asked.

## Supported locales

`en`, `vi`, `ja`, `zh`, `ko` — all five files must be kept in sync at all times.

Locale files: `src/i18n/locales/en.ts`, `vi.ts`, `ja.ts`, `zh.ts`, `ko.ts`

## Never hardcode user-visible strings

Any string a user can read — label, placeholder, button text, toast message, hint, title attribute — must be translated.

```html
<!-- WRONG -->
<button>Add to Cart</button>

<!-- CORRECT -->
<button>{{ t('panel.addToCart') }}</button>
```

## Using t() in components

```ts
import { useI18n } from 'vue-i18n'
const { t } = useI18n()
```

Call `t('namespace.key')` in template and script alike.

## Using t() in composables

`useI18n()` requires a component setup context — it throws outside one. Import the instance directly instead:

```ts
import { i18n } from '@/i18n'
// ...
i18n.global.t('toast.designSaved')
```

## Static label arrays must be computed()

`t()` is only reactive when called inside a reactive context (template or `computed`). Declaring a plain array at the top of `<script setup>` evaluates it once at mount time — locale changes will not update it.

```ts
// WRONG — labels are frozen, locale switch has no effect
const tools = [
  { label: t('tools.text') },
]

// CORRECT — re-evaluated on every locale change
const tools = computed(() => [
  { label: t('tools.text') },
])
```

This applies to: tool label arrays, filter preset name arrays, any array whose elements include translated strings.

## Key namespaces

| Namespace | Contains |
| --- | --- |
| `nav` | TopBar buttons (undo, redo, save, load, …) |
| `tools` | Sidebar and toolbar tool names |
| `panel` | Right-panel and mobile-panel labels, buttons, hints |
| `camera` | Camera modal title, cancel, flip, error |
| `textEdit` | Mobile text edit bar labels |
| `canvas` | Canvas hint overlay |
| `toast` | All toast notification strings |
| `lang` | Language switcher labels |

Add new keys under the closest matching namespace. If no namespace fits, discuss before creating a new one.

## Adding a new string — checklist

1. Pick the right namespace; pick a `camelCase` key.
2. Add the key with an English value to `en.ts`.
3. Add translated values to `vi.ts`, `ja.ts`, `zh.ts`, `ko.ts` in the same commit. Use a placeholder (`'[TODO]'`) only if a translation is genuinely unavailable — never leave a key missing.
4. Use `t('namespace.key')` in the component or composable.
5. If the string is inside an array, ensure the array is `computed()`.

## Locale persistence

`setLocale(code)` in `src/i18n/index.ts` — updates `i18n.global.locale`, writes to `localStorage` (`custommaker_locale`), and sets `document.documentElement.lang`.

Do not write to the locale ref directly anywhere else.

## i18n checklist for every feature / bug fix

After any change that touches UI text:
- [ ] All new strings use `t()` — no hardcoded text in templates or scripts.
- [ ] New keys added to all five locale files.
- [ ] Label arrays that use `t()` are `computed()`.
- [ ] Toast strings in composables use `i18n.global.t()`, not inline string literals.
- [ ] Build passes (`npm run build`).
